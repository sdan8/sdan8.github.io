冲突域

共享式网络中可能会出现信号冲突现象，解决这一问题方法一般是采用载波侦听多路访问/冲突检测技术（Carrier Sense Multiple Access/Collision Detection），简称CSMA/CD。

CSMA/CD工作原理：先听后发，边发边听，冲突停发，随即延迟后重发。

双工模式

  • 半双工（half-duplex mode）：通信双方都能发送和接收数据，但不能同时进行，例如对讲机。此模式下，共享物理介质的通信双方必须采用CSMA/CD来避免冲突。
  • 全双工（full-duplex mode）：通信双方都能同时接收和发送数据，例如电话。此模式下，不会产生冲突，因此不需要使用CSMA/CD机制。

同一链路上相连的两台设备双工模式必须保持一致。

以太网帧结构

分层模型-OSI

国际标准化组织ISO提出。

  • 应用层：最靠近用户的一层，为应用程序提供网络服务。
  • 表示层：提供用户应用层数据的编码和转换功能，确保发送的数据能够被另一个系统的应用层识别。
  • 会话层：负责建立、管理、终止表示层实体的通信会话，该层通信由不同设备的应用程序之间的服务请求和响应组成。
  • 传输层：提供面向连接或非面向连接的数据传递以及进行重传前的差错检测。
  • 网络层：提供逻辑地址，供路由器确定路径。
  • 数据链路层：将比特组合成字节，再将字节组合成帧，使用链路层地址（MAC地址）来访问介质，并进行差错监测。
  • 物理层：设备之间传输比特流，规定了电平、速度、电缆阵脚。

分层模型-TCP/IP

TCP/IP模型不关注底层物理介质，主要关注终端之间的逻辑数据流转发，核心是网络层和传输层。

  • 网络接口层：OSI中的物理层和数据链路层。
  • 网络层：OSI中的网络层。
  • 传输层：OSI中的传输层。
  • 应用层：OSI中的会话层、表示层和应用层。

数据封装

在TCP/IP模型中，数据在传输系统的每一层都将建立协议数据单元（PDU，Protocol Data Unit），不同层的PDU含有不同信息，赋予不同名称。传输层称为Segment（数据段）；网络层称为Packet（数据包）；数据链路层称为Frame（数据帧）。协议栈逐层向下传递数据，并添加报头报尾的过程称为封装。

数据包在以太网物理介质上传播之前必须封装头部和尾部信息，封装信息决定了数据如何传输。

帧格式

以太网使用两种标准帧格式：第一种是DIX v2格式，即Ethernet II格式；第二种是IEEE 802.3格式。两种格式主要区别为Type字段，当Type字段值≤1500（0x05DC）时，使用IEEE 802.3格式；当Type字段值≥1536（0x0600）时，使用Ethernet II格式。以太网中大多数使用Ethernet II格式。

Ethernet II帧格式

  • D.MAC（Destination MAC）是目的MAC地址，6字节。
  • S.MAC（Source MAC）是源MAC地址，6字节。
  • Type是类型字段，用于标识数据字段中包含的高层协议，2字节。取值0x0800的帧代表IP协议帧；取值0x0806的帧代表ARP协议帧。
  • Data是数据字段，是网络层数据，最小长度必须为46字节以保证帧长至少64字节，数据字段最大长度为1500字节。
  • FCS是循环冗余校验字段，提供错误检测机制，4字节。

IEEE802.3帧格式

格式与Ethernet II类似。Length字段把Ethernet II帧的Type字段取代，并且占用了Data字段的8字节作为LLC和SNAP字段。

  • Length字段定义了Data字段包含的字节数。
  • LLC（Logical Link Control）逻辑链路控制，由目的服务访问点DSAP（Destination Service Access Point）、源服务访问点SSAP（Source Service Access Point）和Control字段组成。
  • SNAP（Sub-network Access Protocol）由机构代码（Org Code）和类型（Type）字段组成。Org Code三个字节都为0。Type字段含义与Ethernet II帧中的Type字段相同。根据DSAP和SSAP字段取值又可分为以下几类：
    1. 当DSAP和SSAP都取值0xff时，802.3帧就变成了Netware-ETHERNET帧，用来承载NetWare类型数据。
    2. 当DSAP和SSAP都取值0xaa时，802.3帧就变成了ETHERNET_SNAP帧，可以用于传输多种协议。
    3. DSAP和SSAP其他取值均为纯IEEE802.3帧。

以太网的MAC地址

以太网在二层链路上通过MAC地址（长度48位）来唯一标识网络设备，实现局域网上网络设备之间的通信。

MAC地址由两部分组成：供应商代码和序列号。其中前24为代表供应商代码，由IEEE管理和分配，剩下24为序列号由厂商自己分配。

组织唯一标识符（OUI，Organizationally Unique Identifier）

数据帧的发送和接收

帧从主机发送出后，通过传输介质到达目的端。目的端主机会检查帧头中的目的MAC地址

  • 如果目的MAC地址不是本机MAC地址，并且也不是本机侦听的组播或广播MAC地址，则丢弃该帧。
  • 如果目的MAC是本机MAC地址，则接收该帧，并检查帧校验序列（FCS）字段，和本机计算出的值进行对比来确定帧在传输过程中是否保持完整性。
    • 如果帧的FCS值和本机计算值不同，认为帧已被损坏，丢弃该帧。
    • 如果通过了FCS校验，根据帧头部的Type字段确定将帧发送给上层哪个协议处理。本例中Type为0x0800，表明交给IP协议处理，发送前帧头部和尾部会被剥掉。

IP编址

IP报文头部

  • IP报文头部用来指导网络设备对报文进行路由和分片。
  • 长度为20~60字节。
  • 源和目的IP地址是分配给主机的逻辑地址，可以判断目的端和发送端是否位于同一网段，如果不在同一网段，则需要采用路由机制进行跨网段转发。

IP地址

IPv4地址为32比特的二进制数，通常用点分十进制标识，用来标识网络中的设备。分为网络位和主机位。除网络地址和广播地址意外的其他IP地址都可以作为网络设备的IP地址。

  • 主机位为全0时，表示网络地址。
  • 主机位为全1时，表示广播地址。

IP地址类型

  • 私有地址范围
    • 10.0.0.0~10.255.255.255
    • 172.16.0.0~172.31.255.255
    • 192.168.0.0~192.168.255.255
  • 特殊地址
    • 127.0.0.0~127.255.255.255
    • 0.0.0.0
    • 255.255.255.255

私有地址不会出现在公网中，用于内部使用。127.0.0.0网络中的地址为环回地址，用于诊断网络是否正常。0.0.0.0用于标识任何网络。255.255.255.255是0.0.0.0网络中的广播地址。

子网掩码

用于区分网络部分和主机部分，表示方法和IP地址相同。子网掩码中的1表示网络位，0表示主机位。

每类IP地址都有缺省子网掩码：

  • A类地址缺省子网掩码8位
  • B类地址缺省子网掩码16位
  • C类地址缺省子网掩码24位

变长子网掩码（VLSM）

可以通过改变子网掩码位数，将网络划分为多个子网，缓解了使用缺省子网掩码导致的地址浪费问题。借用主机位作为子网位。

无类域间路由（CIDR）

将路由表中的若干条路由汇聚为一条路由，减少路由表的规模，提高了路由器的可扩展性。

网关

报文在转发过程中，需要确定目的主机与源主机是否在同一网段，如果不在同一网段，则报文需要先转发到网关，然后通过网关转发到目的网段。

网关是指接收并处理本地网段主机发送的报文并转发到目的网段的设备。为实现此功能，网关必须知道目的网段的IP地址。网关设备上连接本地网段的接口地址即为该网段的网关地址。

IP包分片

网络中转发的IP报文长度可以不同，但如果报文长度超过了数据链路所支持的最大长度（MTU），则报文就需要分割成若干较小的片段才能在链路上传输。分割过程叫做分片。

接收端根据报文的标识符（Identification），标志（Flags），及片偏移（Fragment Offset）字段对分片报文进行重组。标识符用于识别属于同一个数据包的分片。标志字段用于判断是否已经收到最后一个分片，最后一个分片的标志字段设置为0，其他分片的标志字段设置为1。

生存时间

报文在转发过程中，如果设备路由规划不合理，可能出现环路，导致报文在网络中无限循环，无法到达目的端。随着这种报文增多，网络将会发生拥塞。为避免环路导致拥塞，IP报文头部包含一个TTL（Time To Live）字段。报文每经过一台三层设备，TTL值减一。初始TTL值由源端设备设置。当报文中TTL降为0时，报文会被丢弃。同时，丢弃报文的设备会根据报文头中的源IP地址向源端发送ICMP错误消息。

协议号

目的端网络层在接收并处理报文以后，需要决定下一步对报文如何处理。IP报文头中的协议字段标识了将会继续处理报文的协议。该字段可以标识网络层协议，如ICMP（Internet Control Message Protocol，因特网控制报文协议），也可以标识上层协议，如TCP（Transmission Control Protocol，传输控制协议，对应值0x06）、UDP（User Datagram Protocol，用户数据包协议，对应值0x11）。

ICMP协议

ICMP是TCP/IP协议簇的核心协议之一，它用于在IP网络设备之间发送控制报文，传递差错、
控制、查询等信息。

ICMP差错检测

ICMP Echo Request和ICMP Echo Reply分别用来查询和响应某些信息，进行差错检测。

ICMP Echo消息常用于诊断源和目的地之间的网络连通性，同时还可以提供其他信息，如报文往返时间等。

ICMP错误报告

ICMP定义了各种错误消息，用于诊断网络连接性问题，根据这些错误消息，源设备可以判断出数据传输失败的原因。比如，如果网络中发生了环路，导致报文在网络中循环，且最终TTL超时，这种情况下网络设备会发送TTL超时消息给发送端设备。又比如如果目的地不可达，则中间的网络设备会发送目的不可达消息给发送端设备。目的不可达的情况有多种，如果是网络设备无法找到目的网络，则发送目的网络不可达消息；如果网络设备无法找到目的网络中的目的主机，则发送目的主机不可达消息。

ICMP数据包格式

ICMP消息封装在IP报文中。ICMP消息的格式取决于Type和Code字段，其中Type字段为消息类型，Code字段包含该消息类型的具体参数。后面的校验和字段用于检查消息是否完整。消息中包含32比特的可变参数，这个字段一般不使用，通常设置为0。在ICMP Redirect消息中，这个字段用来指定网关IP地址，主机根据这个地址将报文重定向到指定网关。在Echo请求消息中，这个字段包含标识符和序号，源端根据这两个参数将收到的回复消息与本端发送的Echo请求消息进行关联。尤其是当源端向目的端发送了多个Echo请求消息时，需要根据标识符和序号将Echo请求和回复消息进行一一对应。

ICMP消息类型和编码类型

类型 编码 描述
0 0 Echo Reply
3 0 网络不可达
3 1 主机不可达
3 2 协议不可达
3 3 端口不可达
5 0 重定向
8 0 Echo Request

ICMP定义多种消息类型

  • 有些消息不需要Code字段来描述具体类型参数，仅用Type字段表示消息类型。比如，ICMP Echo回复消息的Type字段设置为0。
  • 有些ICMP消息使用Type字段定义消息大类，用Code字段表示消息的具体类型。比如，类型为3的消息表示目的不可达，不同Code表示不可达的原因，包括目的网络不可达（Code=0）、目的主机不可达（Code=1）、协议不可达（Code=2）、目的TCP/UDP端口不可达（Code=3）等。

ICMP应用-Ping

Ping是检测网络连通性的常用工具，常用配置参数说明如下：

  • -a source-ip-address指定发送ICMP ECHO-REQUEST报文的源IP地址。如果不指定源IP地址，将采用出接口的IP地址作为ICMP ECHO-REQUEST报文发送的源地址。
  • -c count指定发送ICMP ECHO-REQUEST报文次数。缺省情况下发送5个ICMP ECHO-REQUEST报文。
  • -h ttl-value指定TTL的值。缺省值是255.
  • -t timeout指定发送完ICMP ECHO-REQUEST后，等待ICMP ECHO-REPLY的超时时间。

Ping命令的输出信息中包括目的地址、ICMP报文长度、序号、TTL值以及往返时间。序号是包含在Echo回复消息（Type=0）中的可变参数字段，TTL和往返时间包含在消息的IP头中。

ICMP应用-Tracert

Tracert基于报文头中的TTL值来逐跳跟踪报文的转发路径。为了跟踪到达某特定目的地址的路径，源端首先将报文的TTL值设置为1.该报文到达第一个节点后，TTL超时，于是该节点向源端发送TTL超时消息，消息中携带时间戳。然后源端将报文的TTL值设置为2，报文到达第二个节点超时后，依旧返回TTL超时信息，以此类推，直到报文到达目的地。这样就可以跟踪到经过的每个节点，并计算往返时间。

常用配置参数说明如下：

  • -a source-ip-address指定tracert报文的源地址。
  • -f first-ttl指定初始TTL。缺省是1。
  • -m max-ttl指定最大TTL。缺省是30。
  • -name使能显示每一跳的主机名。
  • -p port指定目的主机的UDP端口号。

ARP协议

地址解析协议（ARP，Address Resolution Protocol），将IP地址和MAC地址进行关联。每一个网络设备在数据封装前都需要获取下一跳的MAC地址，IP地址由网络层提供，MAC地址由ARP协议通过目的IP地址获取，从而实现数据链路层的可达性。

  • Hardware Type表示硬件地址类型，一般为以太网
  • Protocol Type表示三层协议地址类型，一般为IP
  • Hardware Length表示MAC地址长度，单位字节
  • Protocol Length表示IP地址长度，单位字节
  • Operation Code指定ARP报文类型。
  • Source Hardware Address指的是发送ARP报文的设备MAC地址
  • Source Protocol Address指的是发送ARP报文的设备IP地址
  • Destination Hardware Address指的是接收ARP报文的设备MAC地址
  • Destination Protocol Address指的是接收ARP报文的设备IP地址

ARP请求响应过程

每个网络设备一般都有自己的一个ARP缓存表，用来存放IP地址和MAC地址的映射关系。本例中，主机A的ARP缓存表中不存在主机C的MAC地址，所以主机A会发送ARP Request来获取目的MAC地址。ARP Request报文封装在以太帧里。帧头中的源MAC地址为发送端主机A的MAC地址。此时，由于主机A不知道主机C的MAC地址，所以目的MAC地址为广播地址FF-FF-FF-FF-FF-FF。ARP Request报文中包含源IP地址、目的IP地址、源MAC地址、目的MAC地址，其中目的MAC地址的值为0。ARP Request报文会在整个网络上传播，该网络中所有主机包括网关都会接收到此ARP Request报文。网关将会阻止该报文发送到其他网络上。

所有的主机接收到该ARP Request报文后，都会检查它的目的协议地址字段与自身的IP地址是否匹配。如果不匹配，则该主机将不会响应该ARP Request报文。如果匹配，则该主机会将ARP报文中的源MAC地址和源IP地址信息记录到自己的ARP缓存表中，然后通过ARP Reply报文进行响应。

主机C会向主机A回应ARP Reply报文。ARP Reply报文中的源协议地址是主机C自己的IP地址，目标协议地址是主机A的IP地址，目的MAC地址是主机A的MAC地址，源MAC地址是自己的MAC地址，同时Operation Code被设置为Reply。ARP Reply报文通过单播传送。

主机A收到ARP Reply以后，会检查ARP报文中目的MAC地址是否与自己的MAC匹配，如果匹配，ARP报文中的源MAC地址和源IP地址会被记录到主机A的ARP缓存表中。

ARP代理

在上述例子的组网中，主机A需要与主机B通信时，目的IP地址与本机的IP地址在同一个网段，所以A将会以广播形式发送ARP Request报文，请求主机B的MAC地址。但是，广播报文无法被路由器转发，所以主机B无法收到主机A的ARP请求报文，当然也就无法应答。

在路由器上启用代理ARP功能，就可以解决这个问题。启用代理ARP后，路由器收到这样的请求，会查找路由表，如果存在主机B的路由表项，路由器将会使用自己的G0/0/0接口的MAC地址来回应该ARP Request。主机A收到ARP Reply后，将以路由器的G0/0/0接口MAC地址作为目的MAC地址进行数据转发。

免费ARP

免费ARP一般用来探测IP地址是否冲突，当主机分配了IP地址或者IP地址发生变更之后，需要检测这个IP地址在网络上是否是唯一的，避免地址冲突，这时通过发送ARP Request报文来进行地址冲突检测。

主机A将ARP Request广播报文中的目的IP地址字段设置为自己的IP地址，且该网络中所有主机包括网关都会收到此报文。当目的IP地址已经被一个主机或网关使用时，该主机或网关就会回应ARP Reply报文，主机A从而判断出IP地址冲突了。

传输层协议

TCP

TCP协议是面向连接的传输层协议，提供可靠的传输服务，位于TCP/IP模型的传输层。

TCP端口号

协议 端口号
FTP 20、21
HTTP 80
Telnet 23
SMTP 25

每台主机可以拥有多个应用端口，端口号、源IP、目的IP的组合唯一标识了一个会话。端口分为知名端口和动态端口。知名端口范围0~1023，动态端口范围1024~65535。

TCP头部

  • Source Port（源端口号，16位）：源主机应用程序使用的端口号。
  • Destination Port（目的端口号，16位）：目的主机应用程序使用的端口号。
  • Sequence Number（序列号，32位）：用于标识从发送端发出的不同TCP数据段的序号。数据段在网络中传输时，它们顺序可能会变化，接收端依据此序列号可以进行排序。
  • Acknowledge Number（确认序列号，32位）：用于标识接收端确认收到的数据段。确认序列号为成功收到的数据序列号加1。
  • Header length（头部长度，4位）：表示头部占32bit字的数目，它能表达的TCP头部最大长度为60字节。
  • Resv.（保留位，6位）：保留为今后使用，目前应置为0。
  • URG（紧急指针urgent pointer，1位）：当URG=1时，表示紧急指针有效，说明报文段中由高优先级的数据，应尽快传送。
  • ACK（确认序号，1位）：当ACK=1时，表示确认序列号字段（Acknowledge Number）有效，当ACK=0时，确认序列号字段无效。
  • PSH（推送，1位）：表示接收方应该尽快将报文段交给应用层，不再等待缓存填满后再交付。
  • RST（重置，1位）：当RST=1时，表明TCP连接出现严重差错，必须释放连接，再重新建立连接。
  • SYN（同步，1位）：SYN=1表示这是一个连接请求或连接接受报文。
  • FIN（1位）：表示关闭连接
  • Window（窗口，16位）：表示接收端单次收到的数据大小，窗口大小的最大值为65535字节，通常使用改字段进行流量控制。
  • Checksum（校验和，16位）：校验整个TCP报文段，包括TCP头部和TCP数据。该值由发送端进行计算并记录，由接收端进行验证。

三次握手

  1. 主机A发送一个标识SYN的数据段，表示请求建立连接，序列号seq=a
  2. 服务器回复一个标识SYN+ACK的数据段，序列号seq=b，确认序列号ack=a+1，表示对SYN报文的确认
  3. 主机A发送表示ACK的数据段,序列号seq=a+1,确认序列号ack=b+1,表示对服务器SYN报文的确认

传输过程

TCP协议通过对接收到的报文进行确认技术来确保数据准确接收到。如图所示，服务器A发送确认号为M+1500的报文表示接收到前三个数据段，当数据段N+3没接收成功时，服务器A发送确认号依旧是M+1500，此时主机A会重新发送从M+1500开始的数据段。直到所有数据正常接收。

流量控制

接收设备通过window字段来控制对方发送的数据段长度，window字段值为自己能接收的数据大小。如图所示，服务器A接收到4个1024长度的数据段后，由于缓存区满，第4个数据段被丢弃，此时服务器A会以3073为响应，表示接收到这么多的数据，window为3072，表示缓存区最多能够处理3072字节的数据，所以主机A在发送数据时，会重新发送被丢弃的数据，并且窗口大小为3072。

四次握手

当TCP连接断开时，需要进行四次握手。步骤如下：

  1. 主机A想要终止连接，发送标识FIN、ACK的数据段，序列号seq=a，确认序列号ack=b
  2. 服务器A回复ACK数据段，序列号seq=b，确认序列号ack=a+1，表示对主机A发送的FIN报文的确认。
  3. 服务器A下要终止连接，发送FIN、ACK的数据段，序列号seq=b，确认序列号ack=a+1
  4. 主机A回复ACK数据段，序列号seq=a+1，确认序列号ack=b+1，作为对服务器A的FIN报文的确认。

UDP

当传输的数据可靠性要求不高，但是对传输速度和延迟要求高的情况下，可以使用UDP协议，UDP在发送数据时，不需要提前建立连接，但是不能保证数据的可靠性，也无法避免收到重复数据的情况。

UDP头部

  • Source port（源端口号，16位）：源主机的应用程序使用的端口号。
  • Destination port（目的端口号，16位）：目的主机的应用程序使用的端口号。
  • Length（UDP长度，16位）：指UDP头部和UDP数据的字节长度，因为UDP头部信息长度为8字节，所以该字段最小值为8。
  • Checksum（校验和，16位）：功能和TCP校验字段相同，对数据进行校验，该字段可选。

传输过程

发送方在发送数据包时，这些数据包是以有序的方式发送到网络中的，但是每个数据包是独立的在网络中进行传输，不能保证到达接收方时的顺序。因为UDP数据包没有序号，所以接收方无法通过UDP协议对数据包进行排序重组，所以需要应用程序来提供到达确认、排序、流量控制功能。通常情况下，UDP采用实时传输机制和时间戳来传输语音和视频数据。

UDP适合传输时延敏感的流量，如语音视频等。

数据转发过程

  1. 应用程序产生数据，交给传输层进行封装，填写头部信息。
  2. 传输层封装完毕后交给网络层封装，填写头部信息。
  3. IP封装完毕后，开始查找路由，主机必须由到达目的端的路由。
  4. 通过ARP缓存表来找到下一跳的MAC地址。如果没有下一跳的MAC地址，则发送ARP请求。
  5. 数据链路层以太网封装，通常使用的是Ethernet II帧格式，填写源目MAC地址和Type字段，由于上层协议采用IP协议，所以Type字段填充为0x0800。
  6. 主机工作在半双工状态下，所以会使用CSMA/CD来检测链路状态是否空闲。
  7. 如果空闲，发送前导码和定界符附加到帧头部进行传输。前导码用于使接收者进入同步状态，7个字节2进制数，01010101……交替。定界符1字节长度（11010101），用于指示帧的开始。
  8. 数据帧发送到共享以太网中，同一冲突域中的所有设备都会接收到该帧，设备会对帧进行FCS校验，确保帧是完整的，如果不完整，进行丢弃。之后检查目的MAC地址，判断是否和本机符合，如果不符合则丢弃。
  9. 接收设备收到帧后，剥去帧头和帧尾，根据Type字段0x0800上交给上层（网络层）协议进行处理，这里交给IP协议。
  10. 根据IP头部信息进行校验，确保数据完整。根据目的IP地址和路由表项，判断是否可以转发到目的端。并且对TTL进行处理。网络层对IP进行重封装。
  11. 到达数据链路层后，会重新封装一个新的目的MAC地址和源MAC地址，MAC地址可以通过ARP来获得，封装完成后进行发送，交给下个设备，重复10、11步，直到下一跳为目的IP所在的网络。
  12. 数据帧发送到共享以太网中，同一冲突域的设备都会收到（比如有多台服务器），设备进行帧校验，然后检查目的MAC地址是否符合本机，如果符合继续处理该数据帧。
  13. 交给网络层进行解封装，发现目的IP和本机IP一致，剥掉IP头部，交给上一层协议TCP进行处理。
  14. TCP接收到后会判断源目的端口号，比如目的端口号为80，表示HTTP协议，将TCP头部剥离，交给HTTP协议进行处理。
